# Recipe Crafting System — Client UI Implementation Plan (Plan 3 of 3)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire the Vue 3 / TypeScript client so players can buy recipes at a neutral Recipe Shop, craft items at an Artificer, and have account-unlocked recipes flow back into the match — completing the end-to-end recipe crafting loop the Plan 2 server engine already supports.

**Architecture:** The client is a view of server state. There are **no new Vue panels** — the recipe UI surfaces as bottom-action-bar actions generated by `getBuildingActions()` from building capabilities (exactly like the existing shop `buy-item-*`/`reroll-shop` actions). A Recipe Shop (capability `recipe-purchase`) yields `buy-recipe-*` actions from its `recipeInventory`; an Artificer (capability `crafting`) yields `craft-*` actions for the player's unlocked recipes with ingredient have/need computed from the Vault snapshot. Clicking an action dispatches a `purchase_recipe` / `craft_item` websocket command. The account-wide unlock rides the existing profile→join path: `knownRecipeIds` flows profile-API → `useProfile` → `useGameClient` → `GameClient` → `NetworkClient` → `JoinMatchMessage`, mirroring `acquiredAdvancementIds`.

**Tech Stack:** Vue 3, TypeScript, Vite, **vitest** (`npm test` → `vitest run`). Build/type-check: `npm run build` (`vue-tsc -b && vite build`). Work from `client/src/game-portal/`.

**Scope note (final plan of 3):** Plan 1 (elemental items) and Plan 2 (server crafting engine) are complete on this branch. This plan is client-only; the server already exposes `/catalog/recipes`, `recipeInventory` on Recipe Shop building snapshots, `unlockedRecipeIds` on the player snapshot, `knownRecipeIds` on the profile API + `JoinMatchMessage`, and the `purchase_recipe`/`craft_item` commands. Design spec: `docs/superpowers/specs/2026-06-30-equipment-recipe-crafting-design.md`.

## Global Constraints

(from `CLAUDE.md` — every task implicitly includes these)

- **The client is a view of server state.** Never simulate gameplay the server owns. Render `recipeInventory` / `unlockedRecipeIds` / craft results as the server sends them; the server is authoritative (it re-validates every `purchase_recipe`/`craft_item`). Client-side "disabled" states are UX hints only.
- **Custom cursor must always win.** Do NOT write any literal `cursor:` declaration (`pointer`, `default`, `auto`) in component CSS — the global rules in `style.css` handle it. `cursor: not-allowed` is acceptable only for a forbidden/unaffordable state, per-state. This plan adds no new components and should add no `cursor:` rules.
- **`desktopBridge.ts` is the only SPA file that imports `@tauri-apps/api`.** This plan touches none of that; do not probe `window.__TAURI__`.
- **Server naming carried in:** crafted frost variant is `frost_sword`; the Artificer build **hotkey is `a`** (it auto-appears in the worker build menu because the server ships `artificer` with `buildable:true` + `hotkey:"a"` — no client build-menu change needed).
- **Convention:** follow the existing shop wiring (`buy-item-*` / `sendPurchaseItem`) as the reference for every new recipe action/command.

export type RecipeDef = {
  id: string
  name: string
  /** Input item IDs consumed by the craft (2+). */
  inputs: string[]
  /** Gold cost charged at craft time. */

  /** Output item ID produced. */

  if (building.capabilities?.includes('crafting')) {
    // Artificer: one craft action per unlocked recipe. Ingredient counts are
    // computed from the Vault snapshot; a recipe is disabled (not hidden) when
    // the vault lacks the required inputs so the player can see what to gather.
    // Affordability (gold) is NOT gated client-side — the server rejects an
    // unaffordable craft, matching how buy-item actions don't gate on gold.
    const vault = vaultState?.vault ?? []
    const have = new Map<string, number>()
    for (const vi of vault) have.set(vi.itemId, (have.get(vi.itemId) ?? 0) + (vi.stacks ?? 1))
    // Deterministic order: iterate the sorted unlocked ids.
    for (const recipeId of [...unlockedRecipeIds].sort()) {
      const recipe = RECIPE_DEF_MAP.get(recipeId)
      if (!recipe) continue
      const need = new Map<string, number>()
      for (const input of recipe.inputs) need.set(input, (need.get(input) ?? 0) + 1)
      let missing = false
      const parts: string[] = []
      for (const [itemId, count] of need) {
        const owned = have.get(itemId) ?? 0
        if (owned < count) missing = true
        parts.push(`${itemId} ${owned}/${count}`)

        label: recipe.name,

        cost: [{ resourceId: 'gold', amount: recipe.costGold, accent: '#d4a84f' }],
        tooltipTitle: recipe.name,
        tooltipBody: `Craft ${recipe.name}.\nIngredients: ${parts.join(', ')}.` + (missing ? '\n\nMissing ingredients.' : ''),
