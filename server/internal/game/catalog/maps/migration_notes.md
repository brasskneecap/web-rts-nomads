# Map `victoryConditions` Migration Notes

Generated as part of the **campaign-objectives-and-metrics** OpenSpec change.

## Why this file exists

The campaign-objectives-and-metrics change replaces `MapConfig.VictoryConditions`
with `CampaignLevelDef.Objectives` evaluated through the new objective handler
registry. Per the user's design decision (proposal "Map objectives go away.
We will now use campaign objectives as the go to."), the field is being
removed outright — there is no coexistence path.

This file captures, for the record, every condition that was authored on a
map JSON before the migration: what was there, whether it was referenced by
any campaign level, and how it was handled.

## Survey

12 maps live under `server/internal/game/catalog/maps/`. Of those, 6 carried
a `victoryConditions` field. **All 6 had the identical boilerplate shape**:

```json
"victoryConditions": [
  {
    "id": "obj-1",
    "type": "killUnit",
    "count": 1
  }
]
```

A `killUnit` condition is wired via per-unit `metadata.objectiveId` tags — a
unit on the map must declare `objectiveId: "obj-1"` for the condition to ever
complete. **No map's units, buildings, or spawn points reference `obj-1`**
(verified by a repo-wide grep for `objectiveId`, which returned zero matches
under `catalog/maps/`).

The conclusion is unambiguous: **every authored condition was a dead
placeholder generated by the map editor's default**. It was visible in JSON
but could not fire. Victory on these maps was driven entirely by the wave
system (all `totalWaves` cleared) and townhall-destruction fallback — not by
the authored conditions.

## Maps without victoryConditions (no action)

`enemy-test.json`, `enemy-test-small.json`, `battletest.json`, `battletest2.json`,
`grenland-small.json`, `unit-review.json` — all carry no `victoryConditions`
field. No migration needed.

## Behavioural impact

**None expected.** Because the authored conditions were dead (no unit/building
tagged with `obj-1`), Forest play already relied on the wave-survival victory
path. After the migration:

- Custom Game on a stripped map → behaves identically (wave-survival or
  townhall destruction, same as before).
- Forest campaign levels → behave identically up to victory (the legacy
  wave-survival rule still fires). The added `survive_waves` required objective
  in `forest.json` is the registry's expression of the same win condition; it
  AND-gates with the legacy rule per the design's decision on victory rule.

## What changes in maps

The migration mechanically strips the `victoryConditions` field from each of
the 6 affected map JSONs. No other map fields are touched. The catalog loader
(`maps.go`) is updated alongside `protocol.MapConfig` to remove the field
definition, so a future map editor save cannot reintroduce the field
accidentally.

## Future considerations

The single shape `killUnit count:1 unwired` is exactly what the map editor
emits when an author has not actively configured a victory condition. If
custom-game-with-objectives ships later (proposal §"Out of Scope"), the
right place to attach is a new lobby-creation field — NOT a re-introduction
of `MapConfig.VictoryConditions`.

## Follow-up: map-editor-authors-campaign-maps

A follow-up change (the **map-editor-authors-campaign-maps** work) shifts
where per-level campaign data lives:

- **Before:** `catalog/campaigns/<id>.json` carried both campaign metadata
  AND the `levels[]` array (id, display name, prereq, objectives).
- **After:** `catalog/campaigns/<id>.json` is a **header** (id, displayName,
  description, sortOrder, locked). Per-level data — including objectives —
  now rides on the map JSON via an optional `Campaign` block on
  `MapConfig`. Maps with the block are tagged as campaign maps; the
  campaign tree is built by scanning the map catalog at request time.

For Forest, this means:

| Map | Now owns |
|---|---|
| `exploration.json` | `campaign` block for `forest_01` (Forest 1) |
| `exploration2.json` | `campaign` block for `forest_02` (Forest 2) |
| `exploration3.json` | `campaign` block for `forest_03` (Forest 3) |

`forest.json` was trimmed to the header. `swamp.json` was already a header.
Custom Game lobby filters maps with a `Campaign` block out of its dropdown;
the editor's "Load Existing Map" list keeps them so campaign maps are still
editable. To use a campaign map's geometry in Custom Game, duplicate the
map in the editor under a new id without the Campaign block.
