# desktop/

Tauri 2 (Rust) shell for the Nomads desktop build. Owns the application window,
supervises the Go server child process, resolves the per-OS user-data directory,
and (Phase 2) wraps the Steamworks SDK behind a Go-facing JSON-Lines IPC.

This crate is glue. **No game logic lives here.** All simulation, AI, combat,
and lobby state belongs to the Go server.

## Dev workflow

Prerequisites: Rust toolchain (`rustup`), `cargo install tauri-cli --version "^2.0" --locked`,
Visual Studio 2022 Build Tools (Windows).

```sh
# Foundation work — Phase 1 (no Steamworks needed yet)
cd client/src/game-portal
npm run tauri:dev      # starts Vite dev server + builds + launches the shell
```

The shell runs the Go server as a sidecar bundled at
`desktop/src-tauri/binaries/nomads-server-<target-triple>.<ext>`. The Makefile
(§18, not yet written) owns staging the server binary into that location after
`go build -tags embed_spa`. For manual dev iteration:

```sh
cd server && go build -tags embed_spa \
  -o ../desktop/src-tauri/binaries/nomads-server-x86_64-pc-windows-msvc.exe \
  ./cmd/api
```

## Layout

```
desktop/
└── src-tauri/
    ├── Cargo.toml              # crate manifest with exact version pins
    ├── tauri.conf.json         # window, security, bundle, sidecar config
    ├── build.rs                # tauri-build harness
    ├── capabilities/
    │   └── default.json        # explicit IPC + plugin permission allowlist
    ├── icons/                  # generated by `cargo tauri icon <source.png>`
    ├── binaries/               # sidecar Go server binaries (gitignored)
    └── src/
        ├── main.rs             # binary entrypoint
        ├── lib.rs              # tauri::Builder setup + #[command] registration
        ├── supervisor.rs       # §6 — Go child supervisor + ready-line handshake
        └── userdata.rs         # §7 — per-OS user-data dir + writability check
```

`binaries/` is gitignored — the sidecar is built per platform by §18's
packaging pipeline (or by the manual one-liner above for dev). `target/`,
`Cargo.lock` from the workspace is committed (pinned exact versions are
load-bearing).

## Updating Tauri or `steamworks-rs` pins

Per the desktop-shell spec: pins are explicit equalities (`= "x.y.z"`), not
ranges (`^` or `*`). To bump a pin:

1. Edit the version number in `desktop/src-tauri/Cargo.toml`.
2. Run `cargo check` from `desktop/src-tauri/`; address any compilation errors.
3. Run `cargo update` to refresh `Cargo.lock`.
4. In the PR description, explain *why* the bump is happening (CVE, new feature
   we depend on, etc.). Drive-by version bumps are not in scope.

## `steam_appid.txt`

For Steam features in dev (Phase 2), place the appid in
`desktop/steam_appid.txt`. The file is gitignored and is NOT included in
release artifacts (release builds embed the appid via Tauri config /
`SteamAPI_RestartAppIfNecessary`).

## Port collision with `air`

Running `npm run tauri:dev` and `air` simultaneously will both try to bind
`:8080`. Either stop `air` before running the Tauri dev loop, or set
`WEBRTS_PORT` for the Tauri-spawned Go child to a different port via
`tauri.conf.json` env injection (not yet wired — coming with §18).
